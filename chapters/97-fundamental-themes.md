# Appendix: Fundamental Themes

This appendix gathers the durable ideas that run through INFO 5100. They are the habits the course is trying to build underneath the Java syntax.

## 1. You Cannot Verify What You Do Not Understand

AI can produce code faster than a beginning programmer. That does not make the code safe to accept. The student must be able to explain what a component does, why it is designed that way, and what evidence shows that it satisfies the requirement.

## 2. Objects Model Responsibilities

A class is not a convenient file. An object has identity, state, and behavior. Good OO design asks what each object knows, what it can do, and what it must not be responsible for.

## 3. Debugging Is Causal Reasoning

Error messages and AI explanations are aids. Debugging still requires a hypothesis, isolation, evidence, and a test. Guessing is not debugging with confidence; it is debugging without discipline.

## 4. AI Use Requires Phase Gates

The course does not ban AI. It constrains AI. Early modules allow explanation only. Later modules allow scaffolding, then implementation assistance, then collaboration. The student earns delegation by building verification capacity.

## 5. The Project Is One System

The library, inventory, or scheduling application is not fourteen disconnected labs. Decisions compound. A weak class boundary in Module 2 becomes a persistence problem in Module 8 and a defense problem in Module 14.

## 6. Tests Are Evidence, Not Magic

A passing test proves that one stated behavior passed under one condition. It does not prove the whole application is correct. Good engineers know both the value and the limits of tests.

## 7. The Final Defense Is The Point

The final application matters, but the defense reveals whether the student owns it. The course succeeds when a student can show the system, explain the design, identify AI-assisted components, and defend what only the engineer could verify.
